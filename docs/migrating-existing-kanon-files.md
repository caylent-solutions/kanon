# Migrating Existing `.kanon` Files

This guide helps operators who have an existing `.kanon` file -- written by hand
or generated by an earlier `kanon bootstrap` invocation (`kanon bootstrap` was
removed in kanon 3.0.0) -- adopt the `search`, `add`, and lockfile features.

## Audience

You are an operator who already has a `.kanon` file in your workspace. Your file
was created before the `kanon search`, `kanon add`, `kanon remove`, and lockfile
features existed. You want to know what changes with a fresh `kanon install`, and
how to bring the file fully into compliance with the current standard.

If you are starting from scratch, see [list-and-add.md](list-and-add.md) instead.

## First install: lockfile generation

When you run `kanon install` for the first time against an existing `.kanon` file
after upgrading to this release, kanon writes a new file alongside it:

```text
.kanon          <-- your existing file (unchanged)
.kanon.lock     <-- written by kanon install on first run
```

The lockfile captures the resolved git SHAs for every source and every transitive
project at install time. On every subsequent install, kanon reads these pinned SHAs
and installs exactly the same code -- regardless of what new tags or commits have
appeared upstream.

**You must commit both files.** Once `.kanon.lock` exists, your repository has a
reproducible install record. Team members and CI pipelines running `kanon install`
from the same commit will receive byte-for-byte identical checkouts.

Concretely, after the first run:

```shell
# First install against an existing .kanon
kanon install
# kanon prints: "lockfile rebuilt from .kanon (N sources, M projects)"

git add .kanon .kanon.lock
git commit -m "add kanon lockfile"
```

From this point forward, `kanon install` prints:

```text
installing from lockfile (N sources, M projects)
```

To re-resolve all versions (e.g., after bumping a revision spec in `.kanon`),
run:

```shell
kanon install --refresh-lock
```

See [lockfile.md](lockfile.md) for the full lockfile format, conflict-resolution
rules, and refresh options.

## Source-name compliance

### The normalization rule

Every `KANON_SOURCE_<source-name>_*` block in your `.kanon` file uses a
**source name** (the alias) derived from the catalog entry name by this rule:

1. Lowercase the entire entry name.
2. Replace every `-` (hyphen) with `_` (underscore).

No other substitutions are made. The transformation is one-way and lossy.

Examples:

| Entry name (in catalog XML) | Derived source name (in `.kanon`) |
| --------------------------- | --------------------------------- |
| `my-package` | `my_package` |
| `Package-A` | `package_a` |
| `FooBar` | `foobar` |
| `internal_tool` | `internal_tool` (unchanged) |

`kanon add` always writes the normalized source name. If you added entries by hand
or via `kanon bootstrap`, the source names in your file may not match this rule.

### When it matters

A mismatch between the source name in your `.kanon` file and the name kanon would
derive from the catalog entry causes:

- `kanon remove <entry-name>` to report that the source is not found.
- `kanon install --refresh-lock-source <entry-name>` to fail with a lookup error.
- `kanon validate metadata` to emit a `source-name-derivation` warning for the
  affected entry.

### Warning on first install

On the first `kanon install` after upgrading, kanon scans the `KANON_SOURCE_*`
blocks in your `.kanon` file and emits a warning for each block whose key does
not match the normalized form of the corresponding catalog entry name:

```text
WARNING: source name 'Package-A' in .kanon does not match the normalized form
  'package_a' derived from catalog entry 'Package-A'.
  To bring the file into compliance, run:
    kanon remove 'Package-A' && kanon add 'Package-A@<your-spec>'
  Or hand-edit the KANON_SOURCE_Package-A_* keys to use the prefix
  KANON_SOURCE_package_a_*.
  See docs/migrating-existing-kanon-files.md#source-name-compliance.
```

Install continues after the warning. The warning is printed to stderr so it does
not break CI pipelines that capture stdout.

### Remediation path A -- use kanon remove and kanon add

This is the recommended path. It rebuilds the source block from the catalog and
guarantees the normalized source name:

```shell
# Replace Package-A with its normalized equivalent
kanon remove 'Package-A'
kanon add 'Package-A@<your-spec>'
```

After `kanon add`, the `.kanon` file contains `KANON_SOURCE_package_a_*` keys
with the correctly normalized alias. See [list-and-add.md](list-and-add.md) for
the full `kanon remove` and `kanon add` reference.

### Remediation path B -- hand-edit KANON_SOURCE_* keys

If you prefer to edit the file directly, rename the block keys in place:

Before:

```shell
KANON_SOURCE_Package-A_URL=https://example.invalid/org/catalog.git
KANON_SOURCE_Package-A_REF=main
KANON_SOURCE_Package-A_PATH=repo-specs/package-a-marketplace.xml
KANON_SOURCE_Package-A_NAME=Package-A
KANON_SOURCE_Package-A_GITBASE=https://example.invalid/org
```

After:

```shell
KANON_SOURCE_package_a_URL=https://example.invalid/org/catalog.git
KANON_SOURCE_package_a_REF=main
KANON_SOURCE_package_a_PATH=repo-specs/package-a-marketplace.xml
KANON_SOURCE_package_a_NAME=Package-A
KANON_SOURCE_package_a_GITBASE=https://example.invalid/org
```

Rename every key in the block in the same edit -- the required structural keys
(`_URL`, `_REF`, `_PATH`, `_NAME`) plus any optional per-dependency env-var line
(`_GITBASE` above, or any other `${VAR}` name your manifest uses) and the
optional `_MARKETPLACE`. Do not leave any of the old-named keys in the file.

## Common fixes

### No global header is required

There is **no required global header** in the current `.kanon` format. `kanon add`
writes self-contained alias-keyed blocks; when a source's manifest references
`${GITBASE}`, the org base is recorded per dependency in
`KANON_SOURCE_<alias>_GITBASE` (auto-derived from the source URL) rather than a
single global `GITBASE` line -- and a manifest that references no `${VAR}` gets
no env-var line at all. There is also no global `KANON_MARKETPLACE_INSTALL`
toggle: marketplace install is a per-dependency opt-in (see
[Marketplace install is per-dependency](#marketplace-install-is-per-dependency)
below).

The only global that is sometimes required is `CLAUDE_MARKETPLACES_DIR`, and only
when at least one dependency sets `KANON_SOURCE_<alias>_MARKETPLACE=true`:

```shell
CLAUDE_MARKETPLACES_DIR=/path/to/marketplaces
```

- `CLAUDE_MARKETPLACES_DIR` -- absolute path to the directory where marketplace
  plugins are installed. `kanon install` errors only when a marketplace
  dependency is declared but this variable is absent.

### Marketplace install is per-dependency

Earlier kanon used a single global `KANON_MARKETPLACE_INSTALL` toggle. That
variable was **removed**. Marketplace install is now an opt-in per dependency:
add `KANON_SOURCE_<alias>_MARKETPLACE=true` to that dependency's block (absence of
the line is the canonical false; kanon never writes `=false`). Manage the flag
with `kanon marketplace enable <alias>`, `kanon marketplace disable <alias>`, and
`kanon marketplace status`. If your hand-written file still has a global
`KANON_MARKETPLACE_INSTALL` line, delete it and set the per-dependency flag on
each marketplace source instead.

### Non-contiguous source blocks

Each source block consists of the alias-keyed lines:

```shell
KANON_SOURCE_<alias>_URL=...
KANON_SOURCE_<alias>_REF=...
KANON_SOURCE_<alias>_PATH=...
KANON_SOURCE_<alias>_NAME=...
KANON_SOURCE_<alias>_GITBASE=...
```

Hand-written files sometimes interleave comments or blank lines between the
keys of a single source block:

```shell
# Primary catalog
KANON_SOURCE_my_catalog_URL=https://example.invalid/org/catalog.git
# pinned to stable branch
KANON_SOURCE_my_catalog_REF=main
KANON_SOURCE_my_catalog_PATH=repo-specs/my-catalog-marketplace.xml
KANON_SOURCE_my_catalog_NAME=my-catalog
KANON_SOURCE_my_catalog_GITBASE=https://example.invalid/org
```

This is **allowed**. `kanon remove` locates every `KANON_SOURCE_<alias>_*`
line wherever it appears in the file and removes them, preserving the line order
of all remaining content. Non-contiguous blocks do not cause errors.

However, if you want to keep the file tidy for readability, group the three keys
for each source together without intervening lines.

### Stale REPO_URL and REPO_REV legacy env vars

Earlier versions of kanon used a single-source model with two global variables:

```shell
REPO_URL=https://example.invalid/org/old-catalog.git
REPO_REV=main
```

These variables are no longer read by `kanon install`. If `REPO_URL` or
`REPO_REV` are set as environment variables in your shell or CI environment,
kanon prints a warning on install:

```text
REPO_URL and REPO_REV environment variable(s) are deprecated and no longer used
by 'kanon install'. Use --catalog-source to specify a remote catalog source
instead.
```

To migrate:

1. Note the value of `REPO_URL` and `REPO_REV`.
2. Unset `REPO_URL` and/or `REPO_REV` from your shell environment and CI
   pipeline configuration.
3. Add a properly named source block using `kanon add`:

   ```shell
   kanon add '<entry-name>@<your-spec>'
   ```

   Or add the block by hand using the `KANON_SOURCE_<name>_*` format shown above.

See [troubleshooting.md](troubleshooting.md) for the full entry on this warning.

### Mixed line endings

If your `.kanon` file was edited with an editor that writes CRLF (`\r\n`)
line endings -- common when the file originated on a Windows host before
kanon ran under WSL2 -- it may contain a mixture of CRLF and LF (`\n`) line
endings. Windows itself is **not currently supported (planned)**; run kanon
under WSL2 (Windows Subsystem for Linux), where the `.kanon` file behaves
exactly as it does on Linux.

`kanon install` reads the file regardless of line-ending style. `kanon remove`
detects the dominant line ending in the file and preserves it. When the file
contains a mix, kanon normalizes all line endings to LF and prints a warning:

```text
WARNING: .kanon contains mixed line endings; normalized to LF.
```

If you want to fix this proactively, run the following inside WSL2, Linux,
or macOS:

```shell
sed -i 's/\r//' .kanon
```

After normalizing, verify the change with `git diff --check`.

### Missing trailing newline

POSIX requires text files to end with a newline. If your `.kanon` file lacks a
trailing newline, some tools emit warnings and git may display a `\ No newline at
end of file` diff marker.

`kanon install` reads files without a trailing newline without error. `kanon add`
always appends a trailing newline when it writes new source blocks. To fix an
existing file:

```shell
# Append a newline if missing (Linux/macOS)
[ -n "$(tail -c1 .kanon)" ] && echo >> .kanon
```

Verify with `cat -A .kanon | tail -1` -- the last line should end with `$`, not
`^M$`.

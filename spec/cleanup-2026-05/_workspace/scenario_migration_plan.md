## Classification rubric

| Code | Meaning |
|------|---------|
| (a) flag-only | Every `kanon install` / `kanon add` call can take `--catalog-source <url>` directly because the catalog/manifest URL is already in scope at every call site. Single-line edit per invocation. |
| (b) env-only | 5+ install/add call sites all want the same catalog URL; a function-scoped `KANON_CATALOG_SOURCE` fixture is the minimum-diff path. |
| (c) hybrid | Mix: some calls need an explicit flag override while others share the env fixture. |
| (d) genuine deprecation | Test exercises the pre-branch `.kanon`-only flow that the branch converts to a loud `SystemExit(3)`. Must be rewritten to assert `SystemExit(3)` and the WARN text, or (for setup-only callers) replaced with `kanon add` + `kanon install`. |

## Catalog-URL construction convention

For files in classifications (a), (b), (c), the catalog URL is constructed inside
each test's per-function fixture builder as a `file://` URI derived from the bare
manifest repo the test creates (e.g., `manifest_bare.as_uri()`). The canonical
construction expression is:

```
catalog_source = f"{manifest_bare.as_uri()}@<revision>"
```

where `<revision>` is the branch or tag the test already pins (usually `main`).
For files whose tests build a dedicated catalog/content repo with semver tags
(KS, RX, MK, PK, CS), the expression is:

```
catalog_source = f"{content_or_manifest_bare.as_uri()}@main"
# or a PEP 440 constraint where the test already uses one
```
